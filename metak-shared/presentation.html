Distributed Data Replication · Team Brief

How does Zenoh perform —
and where are its limits?

Abstract

We have settled on Eclipse Zenoh as the transport for keeping multiple LAN peers in sync under large, high-frequency change diffs. This study characterizes how it performs and where its limits are — not whether to adopt it. We built six interchangeable implementations of the same single-writer key-value replicator — Zenoh plus five baselines (Custom UDP, Hybrid UDP/TCP, QUIC, WebSocket, WebRTC) — behind one common interface, and run them all through an identical benchmark matrix sweeping rate, workload shape, QoS level, and threading mode.

The baselines aren't candidates — they bracket what's achievable on this hardware so we can read Zenoh's numbers in context: how close it runs to the performance floor (Custom UDP), and how much headroom we trade for its simplicity at our ~100 K updates/sec, sub-10 ms p99 targets.

~5 minutes · study design & results — same-machine + two-machine (WiFi) runs, May 2026

The setup

Two runners, six variants, a five-dimension matrix

  • Two runners (alice, bob), leaderless coordination — same machine or LAN.
  • Six variants of the replicator: Zenoh + 5 baselines.
  • A 5-dimension sweep → 704 spawns in the full config:

variant× rate (vpt × hz)× workload shape× QoS× threading mode

What's a "write"? One update to a single path in the replicated tree — publishing one arora_types::Value (our universal data type: a 35-variant enum of scalars, typed arrays, and nested key-value trees). vpt = how many such values per tick.

  • Rate: vpt × tick_rate_hz = arora_types::Value updates/sec. In scalar-flood (the headline profile) each write is one scalar Value, so 100×100hz = 100 values/tick × 100 = 10 k/s; 1000×100hz = 100 k/s; max ≈ 1 M/s (saturation probe).
  • Workload shapes: scalar-flood (one scalar Value per write), block-flood (each write a fixed block of values), mixed-types (nested scalar+array+struct Values). Same leaf count, different packing.
  • Threading modes: single (sync, WASM-compatible) / multi (per-peer reader threads).
  • Lifecycle per spawn: connect → stabilize 3 s → operate 30 s → silent 3 s. Metrics over operate.
Key clarification: the "100" in 100×100hz is values_per_tick — distinct arora_types::Value updates, one per path. In scalar-flood each is a single scalar (8-byte f64) published on its own path; block/mixed pack the same total leaf count into fewer, composite Value objects. Async-only variants run multi-threaded only.
The lineup

The transport and the bracketing baselines

The transport we're characterizing

  • Zenoh — Rust pub/sub, zero-conf discovery, Zenoh-native QoS. Already chosen — we're mapping its envelope and limits.

Performance floor (reference)

  • Custom UDP — raw sockets + hand-written NACK protocol. The floor: how far Zenoh sits from minimum-overhead is the framework tax we accept.

Bracketing baselines (context, not options)

  • Hybrid UDP/TCP — UDP multicast (QoS 1–2), kernel TCP (QoS 3–4).
  • QUIC (quinn) — multiplexed streams + datagrams + TLS 1.3.
  • WebSocket — TCP + WS framing, reliable QoS (3–4) only.
  • WebRTC — SCTP-over-DTLS-over-UDP, all four QoS levels.

All six implemented and exercised. Aeron evaluated but permanently excluded (Windows C-FFI blocker). Async-only variants (QUIC, Zenoh, WebRTC) run multi-threaded only.

The baselines aren't alternatives we'd switch to — they bracket what's achievable. Custom UDP / QUIC set the floor; Hybrid / WebSocket / WebRTC show what heavier framings cost. They exist so Zenoh's numbers can be read in context.
QoS

Four QoS levels — and the no-skip contract

LvlNameTransportOrderingLoss
1Best-EffortUDP, fire-and-forgetNoneTolerated
2Latest-ValueUDP + seq filterLatest-winsSkipped
3Reliable-UDPUDP + NACKStrictRecovered
4Reliable-TCPTCP (or eq.)StrictKernel-recovered
Strict No-Skip Contract (QoS 3/4): variants must deliver 100 % of accepted writes and block the writer rather than drop. Under overload the failure mode is throughput collapse, not delivery shortfall. QoS 1/2 keep the opposite priority (skip + log).
How to read it: a saturated reliable variant shows up as low throughput, not low delivery.

Each variant runs all four levels — except WebSocket (L3/L4 only, no unreliable mode).

At QoS 3/4 delivery is the priority and the writer blocks under back-pressure; at QoS 1/2 throughput is the priority and the writer skips + logs. Keep this in mind when reading saturation numbers.
Measurement

What we measure

Receive throughput is the headline metric — receivers, not writers, are the sync bottleneck.

Integrity

  • Delivery rate per (writer → receiver) pair.
  • Out-of-order receives; duplicates; unresolved gaps.
  • backpressure_skipped at QoS 3/4 = integrity failure.

Performance

  • Receive throughput (headline); write throughput (context).
  • Latency p50 / p95 / p99 / max, per published write op.
  • Jitter, loss %, connect time, CPU %, memory.

Logging & latency: per-message data is compact Parquet (JSONL is lifecycle-only). Latency = matching each write to its receive by ordering (no seq), exact at QoS 3/4. Cross-machine clocks reconciled via an app-level NTP-style offset exchange (not PTP); single-host runs share one clock.

Mechanics

How a spawn knows it's done

Termination is runner-coordinated and activity-based:

  1. Each variant emits a 1 Hz progress event to stdout (sent, received, phase). The runner reads it.
  2. Runners exchange per-spawn progress over their coordination channel.
  3. When every runner reports its variant has been idle for a few seconds during operate, they agree it's done → silent.
  4. A per-spawn max_spawn_secs wall-clock budget is the safety net.

The variant writes a single eot_sent marker to its log to bound the operate window, so the analysis tool scopes every metric to the same well-defined window across variants.

Why activity-based: it lets each spawn run exactly as long as it needs and terminate cleanly when both peers go quiet, instead of relying on a fixed wall-clock budget. The marker is what keeps the analysis window consistent across variants.

Findings

What we can say about Zenoh

  • Operationally simple — discovery, QoS, recovery all out of the box, no transport code.
  • Delivers 100 % across QoS 1–4 at the realistic 10 k rate — on loopback and on real WiFi.
  • Holds up under stress — 100 % delivery at 100 k writes/s, where WebRTC and single-threaded Hybrid collapse.
  • Most consistent latency in the field — ~8–11 ms mean with a tight spread; steadier than QUIC's jitter (±50–84 ms).
  • Limit — heterogeneous payloads at high rate: the mixed-types workload at 100 k drives Zenoh latency into the seconds (and ~68–70 % delivery on WiFi). Its clearest weak spot.
  • Limit — multi-threaded only: no native single-threaded / WASM path yet (the router-RPC sidecar is the planned route).

Against our targets: at the realistic rate Zenoh sits ~10 ms mean — right around the 10 ms goal (a strict sub-10 ms p99 isn't guaranteed), with full delivery.

This is the heart of the talk: Zenoh is correct, simple, consistent, and robust under stress — a strong result. Be honest about the two real limits: mixed-types at high rate, and no single-threaded mode yet. Neither is a reason to reconsider Zenoh; both are tuning / engineering work.
Replicate it

Run the benchmark yourself — two machines

1. Both machines — clone & build (needs the Rust toolchain, and Python 3.12 with polars + matplotlib):

git clone https://github.com/semio-ai/distributed-data-demos.git cd distributed-data-demos cargo build --release

2. Network — put both on the same subnet. For a WiFi test, leave only the WiFi adapter up — a second active NIC breaks peer discovery:

Disable-NetAdapter -Name "Ethernet" -Confirm:$false # WiFi tests only; Enable-NetAdapter after

3. Run — one machine each, logs to a folder both can write (auto-collects both peers into one run):

# machine A target\release\runner.exe --name alice --config configs\two-runner-all-variants.toml --log-dir z:\shared\ddd # machine B target\release\runner.exe --name bob --config configs\two-runner-all-variants.toml --log-dir z:\shared\ddd

4. Analyze (on either machine, once the run finishes):

python analysis\analyze.py z:\shared\ddd\<run-folder> --summary --dump --diagrams --output z:\shared\ddd\<run-folder>\analysis

Worth comparing across links: run the same matrix over WiFi 2.4 GHz, WiFi 5 GHz, and wired gigabit — the three side by side separate the transport's own behaviour from what the network imposes.

Prereqs: Rust (rustup), Python 3.12 with the analysis deps (polars, matplotlib). configs\two-runner-all-variants.toml runs all six variants; configs\two-runner-zenoh-all.toml is the Zenoh-only subset. The runners agree on a shared run-subfolder name, so pointing both --log-dir at the same network share auto-merges their logs; with no shared drive, use a local --log-dir on each and copy bob's files into alice's run folder before analyzing. Remember to Enable-NetAdapter the Ethernet again after WiFi tests. For the 2.4/5 GHz/wired comparison, just change the link between runs and keep everything else identical.
