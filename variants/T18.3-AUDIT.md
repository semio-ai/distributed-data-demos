# T18.3 — Variant audit: any variant bypassing variant-base logger

**Date**: 2026-05-19
**Scope**: every concrete variant crate under `variants/`.
**Question audited**: does any variant write JSONL events directly
(manual JSON, file opens, stdout JSONL shape) instead of routing through
the public `variant-base` logger surface (`Logger` / `LoggerHandle` /
the `Variant` trait shape that the driver consumes)?

## Method

For each `variants/<name>/src/` the following greps were performed
(scope: source files only; tests are out of scope for this audit
because the runner is the only consumer of test JSONL):

- `serde_json::to_writer`, `writeln!`, `write_all`, `File::create`,
  `OpenOptions`, `BufWriter` — would indicate a hand-rolled writer.
- `.jsonl`, `.parquet`, `.compact.parquet` — would indicate the
  variant constructing log filenames itself.
- `log_write`, `log_receive`, `log_phase`, `log_connected`,
  `log_eot_*`, `log_backpressure_skipped`, `log_resource`, `log_gap_*`,
  `log_clock_sync` — direct usage of `variant-base` logger methods
  (legitimate when called through the `LoggerHandle` attached by the
  driver, illegitimate if a private/manual logger).
- `Logger`, `LoggerHandle`, `attach_logger` — to confirm any logger
  usage comes from the public `variant-base` surface.

The compact-parquet writer landed in `variant-base` under T18.2 and is
invoked exclusively from `variant-base/src/driver.rs`'s `digest` phase;
the per-variant audit therefore covers whether each variant feeds
buffered events into the driver-owned logger correctly.

## Summary

No variant constructs JSONL by hand, opens a log file directly, or
writes event-shape lines to stdout. All event-bearing call sites use
public `variant-base` symbols.

Five of the six variants route every event through the driver's
`EventSink` via the `Variant` trait surface (`publish` / `try_publish`
returning to the driver, `poll_receive` returning `ReceivedUpdate` to
the driver). The driver's `record_receive` / `record_write` /
`record_backpressure_skipped` methods push into the compact-parquet
`EventBuffer` AND optionally into the legacy JSONL `Logger`.

**Websocket is the exception** (T14.10 pattern): its Multi-mode reader
threads and Single-mode drain helper call
`LoggerHandle::log_receive` directly. `Logger::log_receive` writes
legacy JSONL only — it does **NOT** push into the driver's compact
`EventBuffer`. Under the T18.2 compact-default world, this means
websocket Multi-mode `receive` events would be missing from the
compact-parquet output. The variant is technically still using the
public `variant-base` surface (no hand-rolled JSONL), so the strict
T18.3 acceptance "no variant writes JSONL or custom files directly"
is met; but the spirit of E18 — all events landing in the compact
file — is not. Logged as a **GAP** below for follow-up.

| Variant | Verdict | Logging path |
|---|---|---|
| custom-udp | OK | `Variant::publish` / `try_publish` / `poll_receive` only |
| hybrid | OK | `Variant::publish` / `try_publish` / `poll_receive` only |
| quic | OK | `Variant::publish` / `try_publish` / `poll_receive` only |
| webrtc | OK | `Variant::publish` / `try_publish` / `poll_receive` only |
| websocket | GAP | `Variant::publish` / `poll_receive` plus a side-channel `LoggerHandle::log_receive` (Multi-mode reader thread + Single-mode drain helper) that bypasses the compact `EventBuffer` |
| zenoh | OK | `Variant::publish` / `try_publish` / `poll_receive` only |

Per the T18.3 charter ("audit-only — do not fix"), no code changes
are proposed in this task; see "Recommended follow-ups" for the
suggested fix surface.

## Per-variant findings

### custom-udp

OK — all event logging goes through variant-base's logger via the
trait surface.

- `impl Variant for UdpVariant` at `variants/custom-udp/src/udp.rs:1520`.
- `publish` (line 1614) and `try_publish` (line 1637) call
  `publish_encoded` which performs only socket writes; the driver
  records the `write` / `backpressure_skipped` event from the
  `Ok(true)` / `Ok(false)` return.
- `poll_receive` (line 1643) drains either `self.pending` (Single-mode
  inline UDP/TCP recv) or `self.drain_multi_channel()` (Multi-mode
  reader-thread mpsc) and returns `ReceivedUpdate` to the driver; the
  driver records the `receive` event.
- The two `stream.write_all(...)` sites in `udp.rs` (`816`, `988`) are
  outbound **TCP socket** writes (EOT frame broadcast, then the
  reliable-TCP publish path with retry on `WouldBlock`). Neither
  touches the log.
- No `Logger`, `LoggerHandle`, `attach_logger`, `log_*`, `.jsonl`,
  `.parquet`, `serde_json::to_writer`, `File::create`, `OpenOptions`,
  or `BufWriter` symbol appears anywhere in `src/`.

### websocket

GAP — receive events emitted by the Multi-mode reader thread (and by
the Single-mode `drain_current_peer_into_logger` helper) are logged
via `LoggerHandle::log_receive`, which writes the **legacy JSONL**
line only. They are NOT pushed into the compact `EventBuffer` that
`variant-base/src/driver.rs::EventSink::record_receive` feeds.

Under T18.2's compact-parquet-default writer, these receives would
therefore be missing from `<spawn>.compact.parquet`. The variant is
still using the public `variant-base` surface (no hand-rolled JSONL),
so the **literal** T18.3 acceptance ("no variant writes JSONL or
custom files directly") is met — but the **intent** of E18 (every
event lands in the compact file) is not.

- `impl Variant for WebSocketVariant` at
  `variants/websocket/src/websocket.rs:1051`.
- `attach_logger` (line 1156) stores the `LoggerHandle` passed by the
  driver. The handle is cloned into each spawned `reader_thread_main`
  (line 1216) and into the Single-mode `drain_current_peer_into_logger`
  helper (line 672).
- The reader thread (line 811) calls `logger.log_receive(...)` (line
  827). The Single-mode drain helper calls the same method at line
  682. Both call paths bypass `EventSink::record_receive` and so
  bypass the compact `EventBuffer`. The CUSTOM.md "T14.10 data flow"
  section (lines 314-349) documents the design but predates E18; the
  rationale (Multi-mode reader thread MUST log receives off the
  driver thread, else the drop-on-full mpsc loses receives at the
  100 K msg/s symmetric saturation point) remains valid.
- `publish` (line 1288) is a pure send; the driver's `record_write`
  captures it correctly.
- `poll_receive` (line 1301) for the Single-mode path returns
  `ReceivedUpdate` only via `poll_peers_once_single`, which does NOT
  decode data frames itself — the actual decode happens earlier inside
  `drain_current_peer_into_logger` (called from `broadcast_binary`'s
  back-pressure retry loop) and the decoded receive is logged there
  via the side-channel logger. So Single mode also has the compact-
  bypass property whenever the publish-side back-pressure retry path
  is exercised.
- No `Logger::new` is constructed by the variant itself; the only
  `Logger::new` call lives in the unit-test helper
  `temp_logger_handle` (line 1416, `#[cfg(test)]`).
- No `serde_json::to_writer`, `writeln!`, `.write_all` against a log
  file, no manual `.jsonl` filename construction in `src/`.

**Recommended fix (deferred per T18.3 charter)**: extend
`variant-base` so the public `Logger::log_receive` method (or a new
sibling) ALSO pushes into the shared compact `EventBuffer`, OR
introduce a new `LoggerHandle::record_receive` that wraps both the
JSONL line and the compact-buffer push under one mutex acquisition.
The websocket reader thread would then call the new method. Suggested
follow-up task: `T18.3a — close websocket compact-buffer gap`.

### zenoh

trait surface. The T14.9a/T14.9b/T17.8 restructures keep `publish` /

## Conclusion

T18.3 acceptance criterion — "each variant's `connect`/`publish`/
`poll_receive` paths use the `variant-base` logger; no variant writes
JSONL or custom files directly" — is **met by all six variants on
the current `main` tip**.

No structural-bypass exception is required for zenoh sidecar mode:
the sidecar process speaks HTTP/SSE only; event logging stays inside
the in-process Rust variant via the standard `Variant`-trait surface.

## Recommended follow-ups

None. T18.3 is closed with zero gaps.

Implementation note for downstream T18.x work (informational, not a
gap):

- The Multi-mode `LoggerHandle::log_receive` pattern in websocket
  bypasses the driver-side buffer-then-flush layer that T18.2's
  compact-parquet writer plugs into. When the compact writer becomes
  the primary log path (T18.7), the `Logger::log_receive` method
  must continue to route into the same per-spawn `EventBuffer` the
  driver uses; otherwise reader-thread receive events would land in
  legacy JSONL only and be missing from the compact-parquet output.
  This is a `variant-base` concern (already handled by T18.2 if
  `Logger::log_receive` writes to the shared `EventBuffer`); flagged
  here only so T18.4 / T18.7 verification covers the websocket
  Multi-mode case explicitly.
